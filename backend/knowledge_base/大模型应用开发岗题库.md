## 一、Transformer 基础核心

### Transformer 的整体结构是什么？Encoder 和 Decoder 的区别？

整体由 Encoder+Decoder + 多头注意力 + Feed Forward + LayerNorm + 残差连接组成。

- **Encoder**：双向自注意力，主打特征编码/理解，代表模型 BERT
- **Decoder**：掩码自注意力 + 交叉注意力，主打生成，大模型（GPT/LLaMA/Qwen/ChatGLM）都是 Decoder-only 架构

### 为什么 Transformer 用 LayerNorm 而不用 BatchNorm？

- **BN 的问题**：是批次维度归一化，受 batch_size 影响大，NLP 中句子长度不一致，batch_size 难固定
- **LN 的优势**：单样本维度归一化，对每个样本独立归一化，不受 batch 影响，适配变长文本
- **训练稳定性**：LN 能缓解梯度消失，Transformer 的堆叠层多，LN 是刚需

### 多头注意力（MHA）、分组注意力（GQA）、多查询注意力（MQA）的区别？

核心都是优化注意力的显存和速度：

- **MHA**：多头 Q/K/V 独立，效果最好，但显存占用最高，推理最慢
- **GQA**：多个头共享一组 K/V，平衡效果和速度，是目前主流（GPT-4/LLaMA3 都用）
- **MQA**：所有头共享一组 K/V，推理最快，显存最低，效果略降，适合纯推理场景

### FlashAttention 原理？解决了什么问题？

解决标准注意力显存爆炸 + 计算慢的问题：

- **核心原理**：「分块计算 + 重排 + 显存复用」
- **效果**：把原本的 O(n²) 显存开销降为 O(n√n)，同时利用 GPU 的访存特性提速
- **应用**：现在所有大模型训练/推理都用 FlashAttention，7B 模型能省 50% 以上显存

### 常见的位置编码有哪些？各自的优缺点？

- **绝对位置编码**：固定向量，无法处理超长度文本，位置信息泛化差
- **相对位置编码**：编码 token 间的相对距离，泛化性好，但计算复杂
- **RoPE 旋转位置编码**：目前大模型标配，无额外参数，能处理任意长度文本，泛化性拉满（LLaMA/Qwen/ChatGLM 全用）
- **ALiBi**：无位置编码参数，靠注意力偏置实现，训练快，适合小模型

### Transformer 中残差连接的作用？为什么必须加？

解决深层网络梯度消失问题，让梯度能顺利回传。Transformer 一般堆叠几十层，没有残差连接，训练直接发散，无法收敛。

### GELU/ReLU/Swish/GLU 激活函数的区别？为什么大模型用 GELU/GLU？

- **ReLU**：硬截断，梯度容易消失
- **Swish**：平滑版 ReLU
- **GELU**：随机正则化的 ReLU，更贴合自然语言的概率分布，是 GPT/BERT 的标配
- **GLU**：门控线性单元，能筛选有效特征，显存效率更高（LLaMA3/Qwen2 都用 GLU）

### BPE/WordPiece/SentencePiece 分词器的区别？大模型用哪种？

- **BPE**：字节对编码，贪心合并高频子词，GPT 用，分词速度快
- **WordPiece**：基于概率的合并，BERT 用，分词准确率高
- **SentencePiece**：按字符分词，无词汇表限制，适配多语言，ChatGLM/Qwen 用

大模型核心是子词分词，解决 OOV（未登录词）问题。

### Decoder 为什么要加掩码（Mask）自注意力？

因为生成任务是自回归的，要保证生成第 i 个 token 时，只能看到前 i-1 个 token，看不到后面的 token。掩码就是把未来位置的注意力权重置 0，防止信息泄露。

### 为什么 Transformer 能解决 RNN 的长依赖问题？

- **RNN**：串行计算，长文本的梯度容易消失，依赖捕捉能力弱
- **Transformer**：并行计算 + 全局自注意力，能直接捕捉任意两个 token 的依赖关系，无长度限制（理论上）

### 交叉注意力（Cross-Attention）的作用？

Decoder 的核心组件，用 Decoder 的 Q 去匹配 Encoder 的 K/V，让生成的内容能对齐输入的文本特征。比如机器翻译中，生成译文时能对应原文的信息。

### Transformer 的时间复杂度和空间复杂度？

核心是 O(n²×d)，n 是序列长度，d 是词嵌入维度。这也是大模型处理长文本的瓶颈，所以有了 FlashAttention、PageAttention 等优化。

### 为什么大模型都是 Decoder-only，而不是 Encoder-Decoder？

- **场景匹配**：Decoder-only 是自回归生成，更贴合对话/创作/问答等核心场景
- **效率**：训练和推理更高效，不用交叉注意力
- **效果**：同等参数量下，Decoder-only 的生成能力远超 Encoder-Decoder

### Self-Attention 的计算步骤？

三步：

1. 对输入做线性变换得到 Q/K/V
2. 计算 Q 和 K 的相似度得到注意力分数
3. 分数做 softmax 归一化，加权求和 V 得到最终输出

### 为什么要对注意力分数做 Scaled？

因为 Q/K 的维度 d_k 越大，注意力分数的方差越大，softmax 后会出现极端值（要么 0 要么 1），梯度消失。Scaled 就是除以√d_k，把分数归一化到合理区间，让训练更稳定。

### BERT 和 GPT 的核心区别？

- **架构**：BERT 是 Encoder-only（双向），GPT 是 Decoder-only（单向）
- **任务**：BERT 擅长理解类任务（文本分类、相似度），GPT 擅长生成类任务（对话、续写）
- **预训练方式**：BERT 是掩码语言模型（MLM），GPT 是因果语言模型（CLM）

### Transformer 的训练难点有哪些？怎么解决？

难点：显存爆炸、梯度消失、训练速度慢、长文本处理差

解决：

- **显存** → FlashAttention + 量化
- **梯度** → 残差 + LN
- **速度** → 并行计算 + 混合精度训练
- **长文本** → RoPE 插值 + PageAttention

### 什么是自注意力的「注意力坍塌」？怎么解决？

注意力坍塌：注意力分数集中在少数 token 上，模型只关注局部信息，失去全局建模能力

解决：加 dropout、调整 Scaled 系数、用多头注意力分散权重

### 为什么 Transformer 的 FFN 层要把维度先升后降？

升维是为了增加模型的非线性表达能力，降维是为了控制显存和计算量。一般是升维到 4×d_model，这是经验值，效果最优。

### 残差连接的两种方式：Pre-LN 和 Post-LN，区别？

- **Pre-LN**：先做 LN 再做注意力和 FFN，是目前大模型的标配，训练更稳定，收敛更快
- **Post-LN**：先做注意力再做 LN，训练难度大，容易发散，早期 Transformer 用这个

## 二、大模型微调 + 训练

### 大模型的完整训练流程是什么？预训练→SFT→RLHF 分别做什么？

三步核心流程：

1. **预训练**：用海量无标注文本训练，让模型学习语言规律和知识，是基座模型，耗时最久、算力最高
2. **SFT（有监督微调）**：用人工标注的高质量对话数据微调，让模型对齐人类指令，输出符合人类意图的内容
3. **RLHF（人类反馈的强化学习）**：先让人类给模型输出打分，再用奖励模型（RM）训练，最后用 PPO 微调，让模型输出「更优质、更安全」的内容

**核心亮点**：DPO/IPO 是目前替代 RLHF 的主流方法，不用训练奖励模型，步骤更简单，效果持平甚至更好。

### 为什么要做「参数高效微调（PEFT）」？而不是全量微调？

全量微调的问题：

- 显存要求极高，7B 模型全量微调需要 40G+ 显存，普通显卡跑不动
- 训练速度慢，耗时久
- 容易过拟合，泛化性差
- 参数量太大，微调后模型难以部署

**PEFT 的核心**：只训练模型的一小部分参数（1%~5%），冻结基座模型，训练效率拉满，显存需求骤降。

### LoRA/QLoRA/p-tuning v2 原理 + 区别？各自的适用场景？

都是 PEFT 的主流方法：

- **LoRA**：在 Transformer 的注意力层加入低秩矩阵（A+B），只训练这两个小矩阵，其他参数冻结。核心参数 r（秩）、alpha（缩放系数），通用任务 r=8~16，代码生成 r=32~64
- **QLoRA**：在 LoRA 基础上加了 INT4 量化，把基座模型量化成 4 位精度，7B 模型微调仅需 4GB 显存，消费级显卡（RTX3090/4090）就能跑
- **p-tuning v2**：在输入层加入可训练的 prompt 向量，冻结模型参数

**总结**：优先用 QLoRA，兼顾效果和效率。

### LoRA 微调时，过拟合了怎么办？

- **数据层面**：做数据增强（同义词替换、句子重排序）、增加训练数据量、清洗低质量标注数据
- **训练层面**：降低学习率（QLoRA 的最优学习率是 2e-4~5e-4）、增加 dropout（0.1~0.2）、用余弦退火调度学习率、缩短训练轮数
- **模型层面**：减小 LoRA 的 rank 值（比如从 32 降到 8）
- **正则层面**：加 L2 正则，限制参数的取值范围

### 全量微调、LoRA 微调、Prompt Tuning 的优缺点对比？

- **全量微调**：效果最好，缺点是显存高、速度慢、过拟合
- **LoRA/QLoRA**：效果接近全量，显存低、速度快，泛化性好，最优解
- **Prompt Tuning**：显存最低，效果最差，适合小样本场景

### 为什么大模型微调的学习率要设置得很小？

因为基座模型已经预训练了海量数据，参数已经很优，微调只是「对齐人类意图」，不是重新训练。学习率太大会破坏基座模型的知识，导致模型退化。QLoRA 的最优学习率是 2e-4~5e-4。

### 微调时，Batch Size 设置多少合适？

QLoRA 微调 7B 模型，Batch Size 设置 4~8 最优：

- 太小：会导致训练不稳定，梯度波动大
- 太大：显存占用高，且容易过拟合

### 什么是「指令微调」？和普通微调的区别？

- **指令微调**：用「指令 + 输出」的格式做微调，比如「帮我总结这篇文章：xxx」，让模型学会理解人类的指令意图
- **普通微调**：用「输入 + 输出」的格式，比如翻译的「英文 + 中文」

指令微调的模型泛化性更强，能处理多种任务。

### 微调后的模型怎么和基座模型融合？怎么导出部署？

QLoRA/LoRA 微调后得到的是适配器权重（adapter），不是完整模型。融合方式：用 HuggingFace 的 PEFT 库把 adapter 权重合并到基座模型里，生成完整的模型权重。

导出格式：Safetensors（安全、快）或 PyTorch 的 bin 格式。

### 为什么大模型微调要做「梯度累积」？

解决显存不足的问题。把多个小 batch 的梯度累加起来，等效于一个大 batch 的梯度更新，既保证训练效果，又能降低显存占用。梯度累积步数一般设 2~4。

### 混合精度训练（FP16/FP8）的原理？为什么能提速？

把模型的参数从 FP32（单精度）降到 FP16（半精度）：

- 显存占用直接减半
- 计算速度提升 2~3 倍

核心是用 FP16 做前向和反向传播，用 FP32 保存梯度，防止梯度消失。

### 大模型的「知识遗忘」是什么？怎么缓解？

微调后模型在适配特定任务的同时，忘记了基座模型的通用知识，导致通用能力下降。

缓解方法：

- 用小学习率微调
- 加入通用数据做混合训练
- 用 LoRA 只微调部分层，不破坏基座模型

### SFT 的数据集怎么构建？有什么要求？

核心要求：高质量、多样性、对齐人类意图

- **格式**：「指令 + 输入 + 输出」
- **清洗原则**：删除重复数据、低质量标注、敏感内容

### 为什么 DPO 比 RLHF 更受欢迎？

- **RLHF**：需要训练奖励模型 + PPO 微调，步骤多、训练不稳定、容易坍缩
- **DPO**：直接用人类标注的偏好数据训练，一步到位，训练更稳定、速度更快、效果更好

### 微调时，怎么评估模型的效果？

- **客观指标**：困惑度（PPL）越低越好，BLEU（生成）、准确率（问答）越高越好
- **主观指标**：人工评估输出的流畅度、准确性、对齐度

## 三、RAG 检索增强生成

### 什么是 RAG？核心解决大模型的什么问题？

RAG 是检索增强生成（Retrieval-Augmented Generation），核心解决大模型的两大痛点：

1. **幻觉问题**：让模型基于检索到的真实文档生成答案
2. **知识时效性**：RAG 能实时加载最新文档，补充模型的知识盲区
3. **垂直领域适配**：不用微调大模型，就能让模型适配医疗/法律/金融等垂直领域

### RAG 的完整流程是什么？每个环节的核心要点？

标准流程：

1. **文档加载**：支持 pdf/word/txt/网页等格式
2. **文本切分**：最核心的环节，决定检索效果。不能纯按字符切分，要按「语义 + 段落 + 标题」切分，比如切分成 200~500token 的 chunk，保留标题和上下文
3. **Embedding 向量化**：把文本转成向量，选开源的 Sentence-BERT 或 BGE-large-zh
4. **向量库存储与检索**：向量库选 Chroma（轻量）或 Milvus（工业级）
5. **生成阶段**：把检索到的文档拼接成 prompt 上下文，传给大模型

### RAG 的核心痛点是什么？怎么优化？

核心痛点：检索不准、语义割裂、幻觉残留、长文本处理差、召回率低

针对性优化方案：

1. **文本切分优化**：用「语义切分 + 重叠窗口」，每个 chunk 重叠 50token，保留上下文
2. **检索策略优化**：用「混合检索」=稠密检索（Embedding）+ 稀疏检索（关键词）
3. **Embedding 优化**：用「双 Embedding 策略」，或用 Rerank 重排序对检索结果重新打分
4. **Prompt 优化**：限定模型「只基于检索内容回答，没有相关内容就说不知道」
5. **向量库优化**：对高频检索的内容做缓存

### RAG 和大模型微调的区别？各自的适用场景？

- **RAG**：成本低、快速适配新领域，适合知识问答、文档总结
- **微调**：效果更好，适合代码生成、客服对话、需要模型深度理解的场景
- **最优解**：RAG + 微调结合

### 为什么 RAG 会出现「检索到了但回答错了」的情况？怎么解决？

原因：

- 检索到的文档内容不相关，只是相似度高
- 模型的 prompt 理解能力差，没有正确利用检索内容
- 文档内容有歧义

解决：

- 用 Rerank 对检索结果重排序，只保留最相关的前 3 条
- 优化 prompt 模板，明确指令
- 对文档做预处理，清洗歧义内容

### 常用的 Embedding 模型有哪些？怎么选型？

- **BGE-large-zh**：效果最好
- **all-MiniLM-L6-v2**：速度最快、显存最低
- **GPT-4 Embedding**：闭源，效果好

应届生项目用 BGE-large-zh 即可。

### 常用的向量库有哪些？区别？选型依据？

- **Chroma**：轻量级，无依赖，适合本地部署和小体量数据
- **Milvus**：工业级，支持大规模数据和分布式部署
- **Pinecone**：云原生向量库，不用自己部署
- **FAISS**：Facebook 开源，检索速度最快

### RAG 的召回率和准确率怎么评估？

- **召回率**：检索到的相关文档数 / 总相关文档数
- **准确率**：回答正确的问题数 / 总问题数

### 长文本 RAG 怎么优化？比如万字以上的文档

- 做文档分层：把长文档分成「章节→段落→句子」，逐层缩小范围
- 用长文本 Embedding 模型，比如 LongEmbedding
- 用 GPT-4o 这类长文本大模型

### RAG 的工程化落地有哪些注意点？

- 做缓存：用 Redis 缓存高频检索的向量和结果
- 做异步处理：文档加载和向量化用异步任务
- 做监控：监控检索准确率和幻觉率
- 做容错：处理文档加载失败、检索为空的情况

## 四、Agent 智能体

### 什么是大模型 Agent？核心能力是什么？

Agent 是具备感知→思考→决策→执行→反馈能力的大模型智能体，核心是「能自主完成复杂任务」。

核心能力：规划能力、工具调用能力、记忆能力、反思能力。

### Agent 的核心架构是什么？

经典的四模块架构：

1. **感知模块**：接收用户的指令和环境信息
2. **规划模块**：把复杂任务拆解成多个子任务
3. **工具调用模块**：Agent 能调用外部工具（计算器、搜索引擎、代码解释器、API）
4. **记忆模块**：分为短期记忆（缓存最近对话）+ 长期记忆（存储用户的历史需求）
5. **反思模块**：对执行结果做评估，如果失败了，重新调整策略

### Agent 的记忆机制怎么设计？为什么要分短期和长期？

- **短期记忆**：用 Redis 缓存，存储最近 3~5 轮的对话内容，解决「上下文遗忘」问题
- **长期记忆**：用 PostgreSQL + Embedding，把用户的历史需求转成向量存储

核心优势：短期记忆保证对话流畅，长期记忆保证个性化。

### Agent 的工具调用核心逻辑是什么？怎么保证调用的准确性？

工具调用分两步：

1. **意图识别**：Agent 先判断用户的问题是否需要调用工具
2. **工具匹配**：如果需要调用工具，Agent 会选择最合适的工具，并生成工具调用的参数

准确性优化：

- 给工具写清晰的描述
- 对工具调用的结果做校验
- 设置置信度阈值，当置信度低于 0.7 时，追问用户确认

### Agent 和 RAG 的区别？怎么结合使用？

- **RAG**：「检索 + 生成」，主打知识补充，解决幻觉问题
- **Agent**：「思考 + 执行」，主打复杂任务拆解和工具调用

**结合方式**：Agent+RAG 是目前最优的落地方案。

### Agent 的落地难点是什么？怎么缓解？

难点：

- 规划能力不足，复杂任务拆解不清晰
- 工具调用准确率低，容易调用错误的工具
- 记忆能力有限，长对话容易遗忘
- 反思能力差，失败后无法调整策略

缓解：

- 用高质量的 prompt 模板引导 Agent 做规划
- 给工具写清晰的描述
- 优化记忆机制，用 Redis+PostgreSQL 存储记忆
- 对简单任务做规则约束

## 五、大模型部署 + 推理优化

### 大模型部署的核心痛点是什么？怎么解决？

核心痛点：显存占用高、推理速度慢、部署成本高

解决思路：量化 + 剪枝 + 蒸馏 + KV Cache 优化 + 推理框架加速

### 量化的分类？PTQ 和 QAT 的区别？GPTQ/AWQ 的区别？

量化是把模型的 FP32 参数转成低精度（INT4/INT8/FP16）：

- **PTQ（后训练量化）**：训练后量化，不用重新训练模型，直接量化，速度快、成本低。GPTQ/AWQ 都是 PTQ
- **QAT（量化感知训练）**：量化时加入训练，效果更好，但需要算力和数据
- **GPTQ**：按顺序量化，适合 Decoder-only 模型
- **AWQ**：按权重量化，比 GPTQ 速度更快

### KV Cache 是什么？为什么能提升推理速度？

KV Cache 是大模型推理的核心优化。

大模型是自回归生成，生成第 i 个 token 时，需要重新计算前 i 个 token 的 K 和 V。KV Cache 把前 i 个 token 的 K 和 V 缓存起来，生成第 i+1 个 token 时，只需要计算新 token 的 Q，不用重新计算所有 K/V。

**效果**：推理速度提升 3~10 倍，显存占用也降低。

### 大模型推理的延迟和吞吐量怎么优化？

- **量化**：INT4 量化，显存降 4 倍，速度提 2 倍
- **KV Cache 优化**：动态缓存、分页缓存（PagedAttention）
- **推理框架**：用 TensorRT-LLM/TGI/vLLM，比原生 PyTorch 快 3~5 倍
- **批处理**：把多个请求合并成一个 batch
- **投机采样**：用小模型做草稿，大模型做验证

### 7B 模型的部署显存优化方案？

3 个核心优化点：

1. **INT4 量化**：显存从 28GB 降到 7GB
2. **开启 KV Cache 分页**：显存占用再降 20%
3. **用 vLLM 推理框架**：显存占用再降 10%，速度提升 3 倍

最终 7B 模型在消费级显卡（RTX3090/4090）上就能部署，显存仅需 6~8GB。

### 常用的大模型推理框架有哪些？选型依据？

- **vLLM**：目前最优解，速度快、显存低、支持长文本
- **TensorRT-LLM**：英伟达官方框架，速度最快
- **TGI**：HuggingFace 官方框架，易用性高
- **Ollama**：轻量级，一键部署

### 大模型部署的常用格式？ONNX/TensorRT/Safetensors 的区别？

- **Safetensors**：安全、快，适合模型权重存储
- **ONNX**：跨平台，适合推理部署
- **TensorRT**：英伟达的推理引擎，速度最快

### 什么是模型蒸馏？和量化的区别？

- **蒸馏**：把大模型的知识迁移到小模型里，把 70B 模型的知识蒸馏到 7B 模型
- **量化**：降精度
- **区别**：蒸馏是降参数量，量化是降精度。两者可以结合使用

## 六、大模型八股拓展

### 大模型产生幻觉的原因？怎么解决？

原因：

- 预训练知识不完整/过时
- 生成时的自回归特性，模型会编造内容填补空缺
- 训练数据有噪声
- 指令理解偏差

解决：

- RAG 检索增强（最有效）
- 微调对齐（SFT/DPO）
- 优化 prompt，限定模型回答范围
- 加入事实校验模块

### 大模型的长文本处理有哪些方案？

- **RoPE 插值**：把模型的位置编码插值到更长的长度
- **PageAttention**：把 KV Cache 分页，解决长文本显存爆炸
- **滑动窗口注意力**：只关注最近的窗口内的 token
- **文档分块**：把长文档分成多个短块，用 RAG 检索后拼接生成

### 大模型的参数量（7B/13B/70B）的区别？选型依据？

- **7B**：显存低、速度快，适合本地部署和应届生项目
- **13B**：效果比 7B 好，显存要求稍高
- **70B**：效果最好，显存要求极高，适合企业级项目

### 大模型的安全对齐有哪些方法？

- RLHF/DPO：对齐人类偏好
- 指令过滤：过滤敏感指令
- 输出审核：对生成的内容做审核
- 模型微调：用安全数据集微调

### 多模态大模型（文本 + 图像）的核心原理？

核心是「图像 Encoder + 文本 Decoder」，用图像 Encoder 把图像转成向量，和文本向量拼接后，输入到大模型里生成答案。

### 为什么大模型的训练要用千亿级别的数据？

因为大模型是「涌现能力」，参数量越大、数据量越多，模型的理解和生成能力越强。当数据量达到一定阈值后，模型会出现质的飞跃。

## 七、工程类高频题

### 怎么用 Docker 部署大模型项目？

写 Dockerfile，打包项目的依赖（Python、PyTorch、LangChain 等），把模型权重和代码放进镜像，用 docker run 一键启动。

### 大模型的 API 怎么封装？

用 FastAPI/Flask 封装模型的推理接口，提供 POST 请求，输入是用户的问题，输出是模型的回答。加入日志和异常处理。

### 怎么处理大模型的并发请求？

用 Redis 做请求队列，用多线程/多进程处理请求，用批处理提升吞吐量。

### 大模型的日志和监控怎么做？

用 ELK 栈做日志收集，用 Prometheus 做监控，监控显存、CPU、推理延迟、吞吐量。
